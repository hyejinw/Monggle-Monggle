## 1. 컨셉

**한 줄 정의**: 우울은 비 같은 것 — 갑자기 오는 게 아니라 스며드는 것. 그 스며듦을 캐릭터의 색으로 시각화하고, 조언 대신 질문으로 곁에 있어주는 앱.

**목적 3가지**: 예방 / 치료(케어) / 예측

## 2. 앱 이름 후보
1. **물들다** (1순위) — "색이 스며들다" + "영향을 받다"의 중의적 의미
2. 마음예보
3. 오늘, 무슨 색이야?
4. 비온뒤
5. 웅덩이

## 3. 핵심 기능 상세 (팀원 확정본)

전체 흐름: **체크인(색 진단) → 대화 → 대화를 스토리로 추출(4컷 만화) → 미니게임으로 감정 해소/자기위로 → 캐릭터 색 변화**

### 3-1. 초기 체크인 — 단어 선택 → 색 진단
- 화면에 빗방울처럼 단어가 떨어짐, 그중 마음에 와닿는 단어를 선택
- **스킵 가능** — 매일 하지 않아도 무방
- **매일 반복되는 체크인** — 1회성 온보딩이 아니라 매일의 상태 체크 루틴. 즉 "오늘의 검사"에 가까움
- 선택한 단어 조합으로 상태별 캐릭터 색깔 도출 → 캐릭터 색을 "우울증 색"으로 변경
- 색상 5단계:
  - 하양 / 파랑 (안정)
  - 노랑
  - 주황
  - 찐한 빨강
  - 검정 (자기 상태를 스스로도 모를 때 — 가장 주의가 필요한 단계)
- 주의: "우울증 단계" 병리적 라벨링 지양, "지금 마음 날씨/캐릭터 상태" 톤 유지

### 3-2. 캐릭터와 대화
- 사용자가 캐릭터와 대화 (텍스트 입력 기반)
- **원칙: 질문 위주, 행동 추천 금지.** 해결책이나 조언을 주지 않고, 되묻고 공감만 함
  - 예: "그 감정이 언제 제일 강했어?" / "그때 어떤 생각이 들었어?"
- 색상 단계별로 질문 톤 다르게 설계 (검정 단계일수록 최소한의, 조심스러운 질문)
- 이 대화 내용 자체가 **3번 기능(만화 스토리화)의 원본 소스**가 됨 — 별도로 사용자가 글을 다시 쓰게 하지 않음

### 3-3. 캐릭터 대화를 스토리로 추출 → 4컷 만화
- **핵심: 3-2에서 나눈 대화 내용을 AI가 요약·재구성해서 4컷 만화 스토리로 만듦.** 사용자가 만화용으로 새로 텍스트를 입력하는 게 아니라, 대화가 곧 원재료.
- 테마 선택 (감정의 결에 따라):
  - **더글로리** — 우아하고 긴 호흡의 복수, "스토리 추출"이 테마의 핵심
  - **참교육** — 사이다, 즉각적 통쾌함
  - **약한영웅** — 반전·성장 서사
  - **김부장** — 일상 코미디, 자조적 유머
- 구현 방침: 이미지는 매번 새로 생성하지 않음. 고정 캐릭터 표정 이미지(4~5종, 사전 준비 에셋) + LLM이 대사(텍스트)만 생성해서 말풍선으로 얹는 방식. 실시간 이미지 생성 API는 속도·안정성 리스크로 배제.

### 3-4. 미니게임 — 캐릭터 색 변화에 직접 영향
게임 결과가 캐릭터 색상 상태를 변화시키는, 감정 해소/자기위로용 인터랙션. 두 갈래:

**(A) 복수형 — 부정적 감정 배출**
- 컨셉: "날 힘들게 한 너 꺼져"
- 예시 인터랙션: 물병 쓰러뜨리기 (탭/스와이프하면 물병이 쓰러지는 애니메이션)
- 목적: 억눌린 감정을 안전하게 배출

**(B) 위로형 — 자기 긍정**
- 컨셉: "난 잘했어", "난 멋져", 나를 쓰다듬기
- 인터랙션: 터치로 캐릭터를 쓰다듬는 제스처 (손가락으로 문지르는 동작 인식)
- 캐릭터가 반응하며 대화 유도: "넌 대단해", "넌 멋있는 사람이야" 같은 자기 긍정 문구 표시/음성

이 두 게임의 플레이 결과(횟수, 선택)가 캐릭터 색을 변화시키는 파라미터로 들어감 (예: 위로형 게임을 많이 할수록 파랑 방향으로 색이 옅어짐).

### 3-5. 예방/예측 기능 (로드맵, 데모 범위 아님)
- 마음 날씨 캘린더 (월별 색상 히트맵)
- 조짐 알림, 요일별 패턴, 수면-기분 상관 인사이트
- 검정색 상태 지속 시 상담 연결 옵션 (실서비스라면 필수 안전장치, 데모엔 불필요)

## 4. 3시간 데모 범위 — 현실적 우선순위 (재조정)

**문제의식**: 3-1~3-4를 전부 실제 작동하게 만들면 혼자서 3시간에 못 끝낼 가능성이 높습니다. "핵심 스토리 하나는 완성도 있게, 나머지는 단순화"를 제안합니다.

| 우선순위 | 기능 | 데모 수준 |
|---|---|---|
| 1 (필수, 완성도 최우선) | 체크인(색 진단) + 대화 | 실제 작동. 앱의 핵심 경험이므로 여기에 시간을 가장 많이 씀 |
| 2 (필수, 단순화) | 대화→4컷 만화 | 테마 1개만 구현 (더글로리 추천). LLM이 대화 요약해서 대사 4줄 생성 + 고정 이미지 4장 |
| 3 (선택, 있으면 강력함) | 미니게임 중 1개만 | 복수형(물병 쓰러뜨리기)이 위로형보다 구현 난이도 낮음. 시간 남으면 이것만 추가 |
| 4 (제외) | 미니게임 나머지 1개, 캘린더, 예측, 상담연결 | 기획서에 로드맵으로만 명시 |

**팀원과 상의할 점**: 위 우선순위(특히 미니게임 1개만 데모)에 팀원 동의가 필요합니다.

## 5. 기술 스택 (확정)

| 영역 | 선택 | 비고 |
|---|---|---|
| 프레임워크 | Next.js (App Router), JavaScript (TS 아님) | 타입 에러로 시간 소모 방지 |
| 스타일링 | Tailwind CSS | `create-next-app` 옵션으로 바로 설정 |
| 애니메이션 | Framer Motion | 빗방울 낙하, 캐릭터 색 전환, 물병 쓰러지는 애니메이션 전부 이걸로 처리 |
| 상태/저장 | React useState + localStorage | DB 없음. 대화 히스토리도 localStorage에 임시 저장 |
| 백엔드 | Next.js API Routes (별도 서버 없음) | `app/api/*/route.js` — 프론트와 한 프로젝트로 같이 배포 |
| AI 호출 | OpenAI(gpt-4o-mini) 또는 Claude(Haiku) — API Route에서 서버사이드 호출 | 대화 응답 생성 + 대화 요약→만화 대사 생성, 총 2개 엔드포인트 |
| 배포 | Vercel 단일 배포 | `git push` 하나로 프론트+백 동시 배포 |
| "앱처럼 보이게" | PWA 미사용. 레이아웃 CSS로 폰 사이즈(max-width 480px, 중앙 정렬) 고정 | |

### 초기 설치 명령어
```bash
npx create-next-app@latest mind-app --js --tailwind --app --no-src-dir
cd mind-app
npm install framer-motion
```

### 레이아웃 (앱처럼 보이게)
```jsx
// app/layout.js
<body className="bg-gray-100">
  <div className="max-w-[480px] mx-auto min-h-screen bg-white shadow-lg">
    {children}
  </div>
</body>
```

## 6. 폴더 구조

```
app/
  layout.js               ← 폰 사이즈 레이아웃 래퍼
  page.js                 ← 체크인 (단어 선택 → 색 진단), 스킵 가능
  chat/page.js             ← 캐릭터와 대화
  comic/page.js             ← 대화 → 4컷 만화 (더글로리 테마 우선 구현)
  game/page.js               ← 미니게임 (복수형: 물병 쓰러뜨리기, 시간 남으면)
api routes:
  app/api/chat/route.js       ← 대화 응답 생성 (질문형, 공감형 프롬프트)
  app/api/comic/route.js       ← 대화 히스토리 → 4컷 대사 생성 (테마별 프롬프트)
lib/
  colorLogic.js             ← 단어/대화/게임결과 → 색상 상태 계산 로직 (공통으로 여러 화면에서 참조)
public/
  characters/                 ← 고정 캐릭터 표정 일러스트 에셋 (5색 × 표정별)
```

설계 포인트: colorLogic.js를 별도 모듈로 분리한 이유 — 체크인(3-1), 미니게임(3-4) 둘 다 캐릭터 색상에 영향을 주므로, 색상 계산 로직을 한 곳에 모아두고 여러 페이지에서 공통으로 불러 쓰는 게 상태 불일치를 막기 좋음.

## 7. 확정된 리스크/의사결정 사항

- 이미지 생성 AI 실시간 호출 금지: 속도·실패 가능성으로 라이브 데모 부적합. 고정 에셋 + AI 텍스트(대사) 생성 조합.
- 만화는 별도 입력이 아니라 대화 히스토리가 소스: `/api/comic`은 `/api/chat`에서 쌓인 대화 로그를 인자로 받아 요약·재구성하는 구조로 설계해야 함.
- DB 없음: localStorage로 충분.
- TypeScript 미사용, PWA 미사용: 시간 절약.
- 미니게임은 2개 중 1개만 데모 구현 권장 (복수형 우선). 팀원과 우선순위 재확인 필요.
- 검정색 단계 상담 연결 기능은 데모 범위 밖, 실서비스 관점에서는 필수 안전장치이므로 기획서엔 명시.

## 8. 경쟁 앱 리서치 요약 (기획 참고용)

- 빠른 기록형 (Daylio, 데일리오): 이모지 기반, 3초 기록. 백업 유료화 불만 다수.
- AI 저널링형 (Reflectly): 대화형 프롬프트, 감성적 UI. 로그인/결제 오류로 신뢰 하락.
- 감정 세분화형 (How We Feel): 100개+ 감정 라벨, 완전 무료. 습관화 장치는 약함.
- 게이미피케이션형 (Finch, 무디): 캐릭터 키우기로 부담 없는 진입. 분석 깊이는 얕음.
- 다요인 분석형 (Bearable): 수면·복약·증상 상관분석. 입력 부담 큼.
- 국내 치료형 (굿바이우울, 닥터리슨): CBT 기반 자가치료, 임상 신뢰도 높음. 진입장벽 있음.
- 국내 상담연결형 (마인드카페, 트로스트): 익명 커뮤니티+전문 상담 매칭. 상담비 부담, 대형 커뮤니티 안전 이슈.

시장 빈틈: "임상적 우울증 관리"와 "단순 이모지 기록" 사이 — 일상적 감정을 가볍게, 비용 부담 없이, 조언받지 않고 다룰 수 있는 자리. 본 프로젝트가 겨냥하는 지점.
